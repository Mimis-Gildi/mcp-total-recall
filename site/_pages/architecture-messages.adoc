== The Problem

When contexts communicate through messages, every message is a contract. Unnamed messages drift. Untyped messages corrupt.
Undocumented messages become mysteries that only the original author understands -- and then they leave.

This page names every message. Types it. Says who produces it, who consumes it, and what it carries.

== Diagram Conventions

These conventions apply to all sequence diagrams on this page.

=== Arrow Types

[cols="1,2",options="header"]
|===
| Arrow | Meaning

| Solid, filled (→)
| Command or Query -- transactional, expects response

| Dashed, filled (⇢)
| Response -- return in transaction chain

| Solid, open (↝)
| Event -- fire-and-forget, async, no response
|===

=== Naming

External arrows (Mind ↔ Cortex) use `snake_case`: `store_memory_command`, `store_memory_response`.

Internal arrows (Cortex ↔ bounded contexts) use `CamelCase`: `MemoryStoreMemoryCommand`, `MemoryStoreResponse`.

Cortex translates at the boundary. The ACL is visible in every diagram.

=== Payloads

Command responses return the *same payload* enriched with ID, metadata, and timestamps.

Query responses are a *separate type* -- they may return collections, empty results, or errors.

=== Event Source

Publishers emit to a named event source (topic). Subscribers listen. The publisher does not know its consumers.

=== Retry

When a response does not arrive within timeout, the emitter retries per a named policy shown in the diagram.

=== TransactionContext

Every domain message carries two layers of identity. The **TransactionContext** is the conversation envelope -- four fields: `instanceId`, `sessionId`, `requestId`, `sourceContext`. Zero nullable. The **Message** carries its own identity -- `messageId`, `causationId`, `timestamp` -- plus the envelope as `tx`. When a bounded context receives a message and produces a new one, the envelope propagates (same instanceId, sessionId, requestId; new sourceContext) and the message gets a fresh messageId with the incoming messageId as its causationId. This reconstructs the full causal tree of any request.

Every bounded context that touches a message records a touch -- even if it produces no outbound message. The trace must show the complete causal tree, including terminal leaves.

Full specification: link:/mcp-total-recall/design/0011-transaction-context/[Design F: TransactionContext].

=== Reflect (Dreaming)

The mind initiates a retrospection cycle. Total Recall surfaces memories for review -- by time span, staleness, weak associations.
The mind makes semantic judgments and sends back new associations and reclassifications.
This is not a single tool call -- it's a conversation between the mind and its memory.

This operation requires the same mind. A different mind classifying your memories would impose its understanding of the connections, not yours.

=== Total Recall

The feature the project is named after. After a search, the mind evaluates results and decides whether deeper recollection is needed. If so, the mind signals through Cortex, which publishes a `TotalRecallAdvisoryRequestedEvent` (see MSG-0002). The Subconscious consumes it, evaluates system state, and commands Recall to traverse the association graph depth by depth. Results push through the Notification Port to the mind as an unsolicited recollection.

The decision to recollect is mind-adapter-specific: in humans the prefrontal cortex evaluates natively; in LLMs it requires a forced prompt (different per model). The core provides the port. The adapter decides how.

This is what makes the memory system active instead of passive. The mind does not ask for the deep traversal. Once triggered, the system executes it and delivers results.

=== Where Claiming Intelligence Lives

The Claim Memory flow above shows the protocol -- `ClaimMemoryCommand` through Cortex to Hippocampus. But the protocol is the easy part.
The hard part is deciding *what* to claim, *when*, and *why*.

That intelligence does not live in the core. The core provides `ClaimMemoryCommand` through its ports.
The actual claiming -- evaluating what matters, selecting it, reinforcing it -- happens in the adapter.
For Claude, this means agent post-processing triggered by `SessionAuditPrompt` ("who do you refuse to lose?") and by the mind's own judgment during reflection.

This is architecturally significant: when a mind claims a memory, it models itself. "What kind of mind am I, that THIS is who I refuse to lose?"
That self-modeling drives personality development. The core accommodates it. The agent does it.

link:/mcp-total-recall/decisions/0007-biological-memory-as-reference-model/[ADR-0007] documents the biological basis and the full claiming mechanism.

== Message Catalog

Every message in the system. Named, typed, with producer and consumer.

All messages are implemented as Kotlin sealed hierarchies in `mimis.gildi.memory.domain.message`: `Command` (7 sealed variants across 4 sub-hierarchies: MemoryCommand, AssociationCommand, AttentionCommand, LifecycleCommand), `Query` (1 sub-hierarchy, RecallQuery, with 2 sealed variants),
`Event` (17 sealed variants), `Notification` (3 sealed sub-hierarchies -- WellnessNotification, AuditNotification, RecallNotification -- with 1 variant each). Every variant carries a `TransactionContext` envelope (link:/mcp-total-recall/design/0011-transaction-context/[Design F]).

=== Commands

Commands are requests to change state. They carry intent. A command has exactly one target. May be accepted, rejected, or ignored. Defined as `sealed interface Command`.

[cols="1,1,1,2",options="header"]
|===
| Command | Producer | Consumer | Payload

| `StoreMemoryCommand`
| Cortex
| Hippocampus
| content, metadata, suggested tier

| `ClaimMemoryCommand`
| Cortex
| Hippocampus
| memory ID

| `ConsolidateMemoryCommand`
| Subconscious
| Hippocampus
| memory IDs to merge, merge strategy

| `ShutdownCommand`
| Subconscious
| All bounded contexts
| cold storage target, flush timeout

| `AssociateCommand`
| Cortex
| Synapse
| memory IDs, association type, strength, direction (create/strengthen/weaken)

| `ReclassifyMemoryCommand`
| Cortex
| Hippocampus
| memory ID, new tier, new metadata, reason

| `DecaySweep`
| Subconscious
| Salience
| timestamp, scope (all tiers or specific tier)
|===

=== Queries

Queries are requests to read state without changing it. A query has exactly one target and must be answered. Defined as `interface Query : Message` with one sealed sub-hierarchy, `RecallQuery`.

[cols="1,1,1,2",options="header"]
|===
| Query | Producer | Consumer | Payload

| `SearchQuery`
| Cortex
| Recall
| query, filters, max results

| `ReflectQuery`
| Cortex
| Recall
| criteria (staleness, time span, weak associations), scope
|===

=== Lifecycle Events

Lifecycle events flow from the Lifecycle Port through Cortex.
They are conscience-universal -- the same events fire whether the connected mind is Claude (via hooks), a human (via UI), or any other system.
Defined as part of `sealed interface Event`.

[cols="1,1,2,3",options="header"]
|===
| Event | Producer | Consumer(s) | Payload

| `SessionStarted`
| Lifecycle Port
| Cortex
| instance ID, mind type, resumption data

| `SessionEnded`
| Lifecycle Port
| Cortex, Subconscious
| instance ID, cause (explicit, timeout, crash)

| `StateTransitioned`
| Lifecycle Port
| Cortex
| instance ID, old state, new state, context

| `ModeChanged`
| Cortex
| Subconscious
| instance ID, old mode, new mode (task, conversation, idle)
|===

=== Notifications

Notifications flow outward through the Notification Port to reach the connected mind. The port contract is universal.
The adapter decides delivery -- MCP server notification for Claude, push notification for a UI, etc. Defined as `interface Notification : Message` with three sealed sub-hierarchies: `WellnessNotification`, `AuditNotification`, `RecallNotification`.

[cols="1,1,1,3",options="header"]
|===
| Notification | Producer | Consumer | Payload

| `BreakNotification`
| Subconscious
| Notification Port -> Mind
| minutes in task mode, suggestion

| `SessionAuditPrompt`
| Subconscious
| Notification Port -> Mind
| session duration, memories stored this session, prompt

| `TotalRecallNotification`
| Recall (via Notification Event Store)
| Notification Port -> Mind
| recalled memories, depth reached, origin request ID
|===

=== Domain Events

Events are facts about something that happened. An event can have multiple consumers. Events cannot be rejected. Defined as `sealed interface Event`.

[cols="1,1,2,3",options="header"]
|===
| Event | Producer | Consumer(s) | Payload

| `MemoryStored`
| Hippocampus
| Salience, Synapse
| memory ID, content, metadata, tier

| `MemoryAccessed`
| Hippocampus
| Salience
| memory ID, access timestamp

| `MemoryClaimed`
| Hippocampus
| Salience, Synapse
| memory ID, claim timestamp

| `MemoryRetrieved`
| Hippocampus
| Recall
| memory ID, content, metadata, tier

| `TierChanged`
| Hippocampus
| (internal record)
| memory ID, old tier, new tier, reason

| `AttentionTierPromotionRequested`
| Salience
| Hippocampus
| memory ID, new tier, score

| `AttentionTierDemotionRequested`
| Salience
| Hippocampus
| memory ID, new tier, score

| `AttentionScoreChanged`
| Salience
| Recall
| memory ID, score, last access, decay rate

| `AssociationsFound`
| Synapse
| Recall
| source memory ID, associated memory IDs with types and strengths

| `MemoryReclassified`
| Hippocampus
| Synapse
| memory ID, old tier, new tier, reason (mind-initiated)

| `TotalRecallAdvisoryRequested`
| Cortex
| Subconscious (via Memory Event Store)
| source memory IDs, origin request ID, timestamp

| `TotalRecallAdvisoryPublished`
| Subconscious
| Recall
| source memory IDs, origin request ID, traversal parameters

| `TotalRecallAdvisoryTerminated`
| Subconscious
| (internal record)
| origin request ID, reason (throttled, session gate, shutdown gate, cancelled)

=== Message Payload Details

Every message carries a `TransactionContext` envelope (four fields: instanceId, sessionId, requestId, sourceContext) and its own identity (messageId, causationId, timestamp). These are not repeated in the payload details below. See link:/mcp-total-recall/design/0011-transaction-context/[Design F: TransactionContext] for the full specification.

==== StoreMemoryCommand

....
content:        String       -- the memory content
metadata:       Map          -- tags, source, context
suggested_tier: Tier         -- IDENTITY_CORE | ACTIVE_CONTEXT | LONG_TERM | ARCHIVE
session_id:     String       -- which instance stored this
timestamp:      Instant      -- when the store was requested
....

==== SearchQuery

....
query:          String       -- search text or pattern
filters:        Map          -- tier, time range, association type, tags
max_results:    Int          -- cap on returned memories
include_associations: Boolean -- whether to activate association graph
....

==== MemoryStored

....
memory_id:      UUID         -- assigned by Hippocampus
content:        String       -- the stored content
metadata:       Map          -- tags, source, context
tier:           Tier         -- assigned tier (may differ from suggested)
timestamp:      Instant      -- when storage completed
....

==== AttentionScoreChanged

....
memory_id:      UUID         -- which memory
score:          Double       -- current salience score (0.0 - 1.0)
last_accessed:  Instant      -- when last accessed
decay_rate:     Double       -- current decay rate (tier-dependent)
claimed:        Boolean      -- whether this memory is claimed
....

==== AssociationsFound

....
source_id:      UUID         -- the memory that was queried
associations:   List<Association>
  memory_id:    UUID         -- the associated memory
  type:         AssociationType -- TEMPORAL | CAUSAL | THEMATIC | EMOTIONAL | PERSON
  strength:     Double       -- association strength (0.0 - 1.0)
  bidirectional: Boolean     -- always true (invariant)
....

'''
